4.2 Infrastructure Design (Contabo-Based)
4.2.1 Pod Architecture Concept
Notes: A "pod" is a self-contained server cluster capable of hosting X number of Odoo instances. Modular design allows adding pods as you grow. Initial pod = 1 Contabo VPS, future pods = multiple VPS or dedicated servers.
4.2.2 Initial Pod Configuration (50-100 Users)
4.2.2.1 Server Specifications

Notes:
Contabo VPS M:
CPU: 6 vCPU cores
RAM: 16 GB
Storage: 400 GB SSD
Bandwidth: Unlimited (100 Mbit/s)
Cost: ~€9.99/month (~$11/month)
OR Contabo VPS L (if more headroom needed):
CPU: 8 vCPU cores
RAM: 30 GB
Storage: 800 GB SSD
Cost: ~€14.99/month (~$17/month)
Recommendation: Start with VPS L for better performance margin, transition to Cloud VPS for better performance as you grow.
4.2.2.2 Initial Pod Cost Breakdown

Notes:
Contabo VPS L: $17/month
Cloudflare (CDN, DDoS protection): $0 (free tier) or $20/month (Pro)
Domain (.co.zw or .com): $15/year (~$1.25/month)
Backup Storage (Contabo Object Storage 250GB): €2.99/month (~$3.50/month)
Email Service (SendGrid 100k emails): $19.95/month (or start with free 100/day)
Monitoring (UptimeRobot free + basic logging): $0
Total Initial Pod Cost: ~$42-62/month depending on email/CDN choices
Target: Keep under $50/month initially, scale email/CDN as needed
4.2.2.3 Capacity Planning

Notes:
Odoo Instance Resource Usage (per SME customer):
RAM: ~150-300 MB per instance (idle/light use)
CPU: <0.5 core (burst to 1 core during heavy operations)
Storage: 500 MB - 2 GB (database + filestore)
Initial Pod Capacity:
Conservative: 30-40 SME instances on VPS L (leaving 8GB for system, services, cache)
Aggressive: 50-60 instances if mostly hustler tier (lighter usage)
Strategy: Monitor resource utilization, scale horizontally by adding pods when CPU consistently >70% or RAM >80%

Stateless Design: All session data in Redis, allowing any app server to handle any request
4.4.2 Vertical Scaling Triggers
Notes:

Database server upgrade when query latency >100ms avg or connection pool saturation

4.4.3 Performance Optimization
Notes:
CDN for static assets (Cloudflare)

Redis caching for frequent queries (user sessions, instance metadata)
Lazy loading of Odoo modules (only load what customer uses)
Database query optimization and indexing strategy
4.5 Security Architecture
4.5.1 Network Security
Notes:
Firewall rules (UFW): Only ports 80, 443, 22 (SSH with key-only auth) exposed
Cloudflare DDoS protection
VPN for administrative access (WireGuard)
Internal service communication via private Docker network
Fail2ban for SSH brute-force protection
4.5.2 Application Security
Notes:
JWT token authentication with short expiry (15 min access, 7 day refresh)
Rate limiting (10 req/sec per user, 100 req/sec per IP)
Input validation and sanitization (Pydantic models)
SQL injection prevention (ORM, parameterized queries)
XSS protection (Content Security Policy headers)
Secrets management (environment variables, vault for production)
4.5.3 Data Security
Notes:
Encryption at rest (LUKS disk encryption or PostgreSQL pgcrypto)
Encryption in transit (TLS 1.3, HTTPS only)
Database encryption for sensitive fields (payment tokens, passwords)
Automated backups encrypted with GPG
GDPR/POPIA compliance: data export, deletion, consent management
4.5.4 Container Security
Notes:
Non-root users in containers
Minimal base images (Alpine Linux)
Regular security updates (automated with Watchtower or manual monthly)
Container image scanning (Trivy)
Resource limits (CPU, memory) to prevent DoS
4.6 Backup & Disaster Recovery
4.6.1 Backup Strategy
Notes:
Frequency:
PostgreSQL: Daily full + 15-min WAL archiving
Filestore: Daily incremental, weekly full
Config: Version controlled (Git)
Retention: 30-day rolling, 12 monthly snapshots
Storage: Contabo Object Storage (primary), separate cloud provider (secondary for critical)
Testing: Monthly restore test on staging environment
4.6.2 Disaster Recovery Plan
Notes:
RTO (Recovery Time Objective): 4 hours for full service restoration
RPO (Recovery Point Objective): 15 minutes max data loss
Procedure:
Spin up new Contabo VPS
Restore latest database backup
Restore filestore from object storage
Update DNS to new server IP
Verify all services operational
Runbook: Documented step-by-step DR procedure, tested quarterly

5.4 Unit Economics
5.4.1 Customer Lifetime Value (LTV)
Notes:
Formula: LTV = (Average Monthly Revenue per Customer × Gross Margin) / Monthly Churn Rate
Assumptions:
Avg revenue/customer: $30 (mix of plans, weighted)
Gross margin: 80% (SaaS typical)
Monthly churn: 5% (60% annual retention, improve to 3% over time)
Calculation: LTV = ($30 × 0.80) / 0.05 = $480
Target: Increase to $800+ by Year 3 (lower churn to 3%, upsell to higher plans)
5.4.2 Customer Acquisition Cost (CAC)

Formula: CAC = (Sales + Marketing Costs) / New Customers Acquired
Target: <$100 per customer (payback in 3-4 months at $30/month avg)
Channels:
Organic (SEO, referrals): $20-40 CAC
Paid ads: $80-150 CAC
Partnerships: $60-100 CAC (commission-based)
Blended CAC: $70 (optimistic), $120 (conservative)
5.4.3 LTV:CAC Ratio
Notes:
Current: $480 LTV / $120 CAC = 4:1 (healthy, VC-backable is 3:1+)
Target: 6:1+ by improving retention and reducing CAC via word-of-mouth
5.4.4 Payback Period
Notes:
Formula: CAC / (Monthly Revenue × Gross Margin)
Calculation: $120 / ($30 × 0.80) = 5 months
Benchmark: <12 months is acceptable, <6 months is excellent
Strategy: Encourage annual prepay to reduce payback period to <1 month

Over-provision slightly (better to have 20% headroom than 0%)
